Implementation plan

Pangolin Compose Wiring

Make AegisNode deploy Pangolin, discover application resources from Docker Compose labels, and prove the workflow with an authenticated Beszel and Dozzle stack.

Certificate decision: keep Traefik's existing ACME HTTP-01 resolver. Pangolin will request an individual certificate for each configured hostname. No wildcard certificate, DNS API, or DNS-provider integration is in scope.
Configuration tracking decision: this work does not choose or implement Git, S3, or any other system for storing, versioning, or distributing server and stack configuration. AegisNode will generate and deploy the files needed for this test. The tracking methodology will be designed separately after the Pangolin wiring has been validated.

1. Goals and success criteria

Repeatable deployment

AegisNode generates and deploys the Pangolin, Traefik, Newt, and test stack configuration consistently. Persisting those inputs in a configuration-tracking system is deferred.

No setup UI required

A fresh deployment creates the Pangolin administrator, organization, and local Newt site through Pangolin's HTTP endpoints.

Compose-driven exposure

Adding Pangolin labels to an application service creates or updates its public resource without editing the Pangolin dashboard.

Working test stack

Beszel monitors the VPS, Dozzle displays its container logs, and both are reachable only through Pangolin authentication.

2. Findings and constraints

  • Pangolin blueprints support YAML and flattened Docker labels. Newt can continuously reconcile those labels when Docker discovery is enabled.
  • Pangolin's documented initial setup is UI-based, but the UI uses PUT /api/v1/auth/set-server-admin. AegisNode can call this endpoint with the setup token, then use the normal login API to create the organization and site.
  • This bootstrap API is not a static configuration interface. Pangolin images must be pinned and the endpoint contract must be tested before an upgrade.
  • Directly pre-populating Pangolin's SQLite database is out of scope. It would bind AegisNode to an internal schema and bypass application validation.
  • HTTP-01 requires each public hostname to resolve to the VPS and requires inbound TCP port 80. A wildcard DNS record is convenient but optional; explicit records for Pangolin, Beszel, and Dozzle also work.

4. Runtime configuration model

Files generated and deployed by AegisNode

  • Pangolin Compose file and config.yml.
  • Traefik static and dynamic configuration files.
  • Observability Compose file and Beszel system configuration.
  • Pangolin labels for Beszel and Dozzle.
  • Explicit, tested container image versions.

These runtime files are deployment outputs for the current test. This plan does not require the observability stack or its rendered configuration to be committed to source control.

Secrets

  • Pangolin server secret, setup token, and administrator password.
  • Newt site ID and secret.
  • Beszel initial-user password, system token, and Hub key material.

These values remain in AegisNode's owner-only profile secret file and in required root-owned 0600 runtime files on the VPS.

Application label contract

labels:
  - pangolin.public-resources.beszel.name=Beszel
  - pangolin.public-resources.beszel.mode=http
  - pangolin.public-resources.beszel.full-domain=beszel.${BASE_DOMAIN}
  - pangolin.public-resources.beszel.auth.sso-enabled=true
  - pangolin.public-resources.beszel.auth.sso-users[0]=${PANGOLIN_ADMIN_EMAIL}
  - pangolin.public-resources.beszel.targets[0].hostname=beszel
  - pangolin.public-resources.beszel.targets[0].port=8090
  - pangolin.public-resources.beszel.targets[0].method=http

Dozzle uses the same shape with its own resource key, domain, hostname, and port 8080. Application services use expose, not host ports.

5. Implementation phases

Phase 1 — Extend configuration rendering

  • Extend the existing Pangolin and Traefik configuration generators.
  • Add rendering for the observability Compose and Beszel configuration files.
  • Pin Pangolin, Gerbil, Traefik, Badger, Newt, Beszel, and Dozzle versions.
  • Render validated values and preserve existing file permissions.

Phase 2 — Automate Pangolin bootstrap

  • Generate a Pangolin administrator password and save it in profile secrets.
  • Wait for Pangolin health and check initial-setup status.
  • Create the administrator only when initial setup is incomplete.
  • Log in and idempotently create organization aegisnode.
  • Create Newt site local-vps with stable generated credentials.
  • For an existing registered installation without a saved password, require a one-time PANGOLIN_ADMIN_PASSWORD environment value.

Phase 3 — Add host-level Newt discovery

  • Create external network aegis-public.
  • Run one Newt instance per VPS on that network.
  • Enable Docker discovery and network validation.
  • Put a read-only Docker socket proxy between the Docker daemon and Newt, Beszel, and Dozzle; deny write and exec endpoints.

Phase 4 — Deploy observability stack

  • Add an observability setup stage after proxy.
  • Deploy Beszel Hub, a local Beszel agent, and Dozzle in one Compose project.
  • Preconfigure Beszel's local system and agent token without using its UI.
  • Set Beszel's trusted authentication header to Remote-Email.
  • Set Dozzle's authentication provider to forward-proxy.
  • Disable Dozzle container actions and shell access.
  • Apply Pangolin SSO labels restricted to the initial administrator.

Phase 5 — Update operator workflow

  • Add optional --pangolin-admin-email, defaulting to the existing Let's Encrypt email.
  • Add aegisnode pangolin-credentials --profile <id>.
  • Show the new observability stage in the TUI and run summary.
  • Replace wildcard-certificate guidance with DNS requirements for the three concrete hostnames.

6. Verification and acceptance

Automated checks

  • Unit-test configuration rendering, escaping, validation, and secret generation.
  • Contract-test Pangolin setup, login, organization, and site API responses.
  • Test fresh profiles and migration behavior for existing profiles.
  • Run docker compose config --quiet against rendered fixtures.
  • Run go test ./... and go vet ./....

Disposable VPS smoke test

  1. Point pangolin.example.com, beszel.example.com, and dozzle.example.com at the VPS, either explicitly or through a wildcard DNS record.
  2. Run the complete AegisNode setup twice and confirm the second run is idempotent.
  3. Confirm all infrastructure and observability containers are healthy.
  4. Confirm Newt discovers and applies both Compose-labeled resources.
  5. Confirm each hostname serves a valid individual Let's Encrypt certificate.
  6. Confirm anonymous requests are redirected to Pangolin authentication.
  7. Confirm the administrator reaches Beszel and Dozzle without a second login.
  8. Confirm Beszel reports the VPS online and Dozzle displays container logs.
  9. Confirm ports 8090, 8080, and the Docker proxy are not publicly reachable.

7. Explicit non-goals

  • Choosing or implementing Git, S3, or another configuration-tracking system.
  • Requiring the Beszel/Dozzle test stack to be stored in source control.
  • Wildcard TLS certificates or DNS-01 challenges.
  • Cloudflare or any other DNS-provider API integration.
  • Automatic DNS record creation.
  • Pangolin database pre-population.
  • General-purpose application deployment beyond the first test stack.

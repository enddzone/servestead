Plan overview

Make guided setup one command, profile-aware, and observable.

The current TUI asks for repeated values and runs one selected path at a time. This plan keeps the native Go CLI architecture, adds persistent server profiles, collects all required setup input before remote changes begin, and upgrades setup into a dashboard that executes bootstrap, hardening, networking, and proxy deployment as one observable run.

Answered assumption

Multiple profiles are required. `setup --ip` selects or creates a profile, then defaults to the full end-to-end server setup plan.

Upfront intake

The user only sees fields that cannot be inferred, defaulted, loaded from the profile, or generated safely, and sees them before execution starts.

Verification style

Each implementation phase includes product acceptance criteria plus tests that can run without a live VPS where possible.

Backend Implementation Handoff

Complete backend capabilities

  • `setup --ip` creates or reuses a saved profile and runs bootstrap, harden, network, and proxy as one full setup plan.
  • Profiles are stored under `filepath.Join(os.UserConfigDir(), "aegisnode")` with separate `profile.json`, `state.json`, `secrets.json`, and JSONL run logs.
  • Profile files, secrets, and log directories use owner-only permissions. JSON save paths are atomic and corrupt JSON errors name the affected file.
  • `GenerateServerSecret()` uses 32 random bytes encoded as base64url; profile-aware setup saves and reuses the Pangolin server secret without printing it.
  • `TaskReporter` and `TaskEvent` are wired through bootstrap, harden, network, and proxy stages. Events update run/stage state and append JSONL logs.
  • Second and later runs skip stages already marked complete in any prior run and print explicit skip messages to the user.
  • Direct commands remain compatible, including `proxy --server-secret` and deprecated `--postgres-password` alias.

UI work still needed

  • Replace the legacy mode-first setup TUI with a higher-level app model: profile picker, upfront intake, plan review, dashboard, run view, and failure/retry states.
  • Use `bubbles/list` for profiles and advanced choices, `bubbles/table` for stage state, `bubbles/progress` plus spinner for active runs, `bubbles/viewport` for logs, and `bubbles/help` for key hints.
  • Expose completed/pending/failed/skipped stages from `ProfileState` instead of parsing human-readable output.
  • Show idempotent second-run behavior in the dashboard: completed stages should appear as skipped or complete, not as rerun work.
  • Add an intentional advanced edit path for users who need to change SSH users, private key path, domain, email, skipped stages, or fresh profile creation.
  • Keep non-interactive/script behavior intact: direct commands and `setup --ip --domain ... --email ... --yes` should not depend on the TUI.
Phase 1

Build the Profile Store

Create a small persistence layer before changing the TUI. The store gives every later UX improvement a reliable source of truth for connection values, full-run state, generated secrets, and logs.

Product Behavior

  • `aegisnode setup --ip 203.0.113.10` creates or selects a profile, collects missing values, then builds the full setup run.
  • If more than one profile exists for the same IP, the TUI shows last-used timestamps and asks whether to resume, rerun, or create a fresh profile.
  • Running `aegisnode setup` without `--ip` opens a profile picker before the upfront intake.
  • Old profiles are never overwritten by "start over"; a new profile ID is created.

Implementation Tasks

Acceptance Criteria

  • Profile saves are atomic: write temp file, `fsync` where practical, then rename.
  • Secrets and profile files are created with owner-only permissions, ideally `0600` for files and `0700` for profile directories.
  • Corrupt JSON produces a clear error naming the profile path and does not delete data.
  • The old command runners still accept explicit flags for non-interactive use.

Phase 4

Convert Task Execution to Structured Events

The current `runTasks` function writes simple lines to an `io.Writer`. The TUI needs structured events so it can update full-run progress, save stage state, render logs, and recover cleanly from failures.

Code Direction

type TaskEventType string

const (
    TaskRunStarted   TaskEventType = "run_started"
    TaskStarted      TaskEventType = "task_started"
    TaskLogLine      TaskEventType = "log_line"
    TaskSucceeded    TaskEventType = "task_succeeded"
    TaskFailed       TaskEventType = "task_failed"
    TaskRunCompleted TaskEventType = "run_completed"
)

type TaskEvent struct {
    Type      TaskEventType `json:"type"`
    RunID     string        `json:"run_id"`
    Stage     string        `json:"stage"`
    TaskIndex int           `json:"task_index,omitempty"`
    TaskTotal int           `json:"task_total,omitempty"`
    TaskName  string        `json:"task_name,omitempty"`
    Stream    string        `json:"stream,omitempty"` // stdout or stderr
    Line      string        `json:"line,omitempty"`
    Error     string        `json:"error,omitempty"`
    Time      time.Time     `json:"time"`
}

type TaskReporter interface {
    Report(TaskEvent)
}

func runStageWithReporter(ctx context.Context, client remoteClient, sshUser string, runID string, stage string, tasks []Task, reporter TaskReporter) error {
    reporter.Report(TaskEvent{Type: TaskRunStarted, RunID: runID, Stage: stage, TaskTotal: len(tasks), Time: time.Now()})
    for index, task := range tasks {
        reporter.Report(TaskEvent{Type: TaskStarted, RunID: runID, Stage: stage, TaskIndex: index + 1, TaskTotal: len(tasks), TaskName: task.Name, Time: time.Now()})
        if err := client.Run(ctx, privilegedCommand(sshUser, task.Apply)); err != nil {
            reporter.Report(TaskEvent{Type: TaskFailed, RunID: runID, Stage: stage, TaskName: task.Name, Error: err.Error(), Time: time.Now()})
            return fmt.Errorf("task %q failed: %w", task.Name, err)
        }
        reporter.Report(TaskEvent{Type: TaskSucceeded, RunID: runID, Stage: stage, TaskIndex: index + 1, TaskTotal: len(tasks), TaskName: task.Name, Time: time.Now()})
    }
    reporter.Report(TaskEvent{Type: TaskRunCompleted, RunID: runID, Stage: stage, TaskTotal: len(tasks), Time: time.Now()})
    return nil
}

Implementation Tasks

Acceptance Criteria

  • CLI output remains usable in scripts and tests.
  • The TUI can render progress without parsing human-readable text.
  • Every failed run records the stage, task name, error, and recent log lines.
  • Profile state is updated even when an end-to-end run fails midway.
Phase 5

Generate and Persist Secrets Safely

The current proxy flow asks for `--server-secret` and stores it in a field named `PostgresPassword`. That value is written to Pangolin's `server.secret`. It should be generated once per profile during upfront intake and reused for every proxy deployment on that profile.

Server Secret Decision

The server secret is an application secret for the Pangolin server configuration, not a user-facing account password. The user should not invent it. Generate a high-entropy random value with `crypto/rand`, save it in the profile's secrets file before the full run starts, and pass it into proxy deployment automatically. Keep the existing `--server-secret` flag for scripts and backward compatibility.

Implementation Tasks

Acceptance Criteria

  • A full setup run on a new profile creates the proxy secret automatically before execution.
  • A full rerun or proxy retry on the same profile reuses the same secret.
  • Deleting only `secrets.json` fails with a clear recovery message or regenerates only after explicit confirmation.
  • Tests verify the secret generator length, randomness shape, persistence, and no accidental log exposure.

Code Direction

func generateServerSecret() (string, error) {
    raw := make([]byte, 32)
    if _, err := rand.Read(raw); err != nil {
        return "", err
    }
    return base64.RawURLEncoding.EncodeToString(raw), nil
}

type ProfileSecrets struct {
    ServerSecret string `json:"server_secret"`
}

func (secrets *ProfileSecrets) EnsureServerSecret() error {
    if secrets.ServerSecret != "" {
        return nil
    }
    generated, err := generateServerSecret()
    if err != nil {
        return err
    }
    secrets.ServerSecret = generated
    return nil
}

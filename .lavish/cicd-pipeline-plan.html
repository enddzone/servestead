Recommended defaults

Specific choices I would implement first

Release strategy

Release Please manages semantic versions and changelog generation. GoReleaser and Buildx run only after Release Please creates a release tag.

Feature branch gate

Gate PRs into `main` with required status checks. Also run checks on branch pushes, but avoid protecting every `feature/**` branch unless you explicitly want to block direct branch updates.

Security posture

Use first-party checks where possible: CodeQL, dependency review, artifact attestations, and govulncheck. Use Renovate for daily dependency scans and update PRs. For third-party scanners, pin tightly and keep permissions minimal.

Dependency automation

Prefer Renovate over Dependabot here because it can manage Go modules, GitHub Actions, Docker tags, and release-tool versions from one configurable policy. Daily scans should create PRs, but merges still depend on CI, Sonar, CodeQL, and security checks.

Tradeoffs

Where the design can go two ways

Decision Recommended Alternative
Release timing Release PR merge publishes a version. Every main merge publishes, but that creates noisy versions for docs and tests.
Docker image build Buildx with `Dockerfile.release` for stable multi-arch images. GoReleaser Docker publishing centralizes config, but current Docker support choices need closer version scrutiny.
Scratch base Scratch plus copied CA certs and static binary. Distroless static is easier operationally, but it is not scratch.
Action pinning Pin third-party security actions by SHA and let Renovate open daily update PRs for GitHub Actions. Semver tags are easier to maintain but weaker against supply-chain compromise.
Dependency bot Renovate with daily scheduled scans, Dependency Dashboard, and grouped non-major updates. Dependabot is simpler, but less flexible for grouping, dashboards, and cross-ecosystem policy.
Sonar enforcement Fail the workflow and require the Sonar quality gate check. Report-only Sonar is simpler but does not gate feature branches.

Repo settings

Things YAML cannot fully do

The implementation can add workflows and config, but these settings need repository access or manual confirmation.

Secrets

  • SONAR_TOKEN
  • RENOVATE_TOKEN or a GitHub App token if self-hosting Renovate through GitHub Actions
  • Optional scanner token only if a selected scanner requires one
  • No GitHub token secret needed for normal release and GHCR publishing; use `GITHUB_TOKEN` with scoped permissions

Permissions

  • contents: write for releases
  • packages: write for GHCR
  • security-events: write for SARIF
  • id-token: write and attestations: write for provenance

Rulesets

  • Require PR before merge to `main`
  • Require CI, Sonar, CodeQL, and security checks
  • Block direct pushes to `main`
  • Optionally protect `feature/**` if that is what "gate feature branches" means

Security features

  • Enable GitHub code scanning
  • Enable dependency graph and Dependabot alerts for GitHub-native vulnerability visibility
  • Enable Renovate Dependency Dashboard and daily scans
  • Install or connect SonarQube Cloud GitHub integration
  • Confirm GHCR package visibility

Risk register

Issues to handle explicitly

  • Release Please expects Conventional Commits.

    Without `feat:`, `fix:`, and breaking-change markers, version bumps and changelog sections will be inconsistent.

  • Scratch images do not include CA certificates.

    Servestead talks to cloud APIs over TLS, so the release image should copy the CA bundle into scratch or use an explicitly accepted non-scratch base.

  • Forked PRs and Sonar tokens can conflict.

    If outside contributors are expected, token access and PR analysis mode need careful handling.

  • Third-party scanner actions are a supply-chain surface.

    Pin scanners tightly, keep permissions scoped, and prefer first-party or official tooling where it gives equivalent coverage.

  • Renovate needs a token strategy.

    The GitHub-hosted Renovate app is simpler to operate, but a scheduled GitHub Actions runner gives this repo direct daily scan cadence. A self-hosted runner should use a scoped token and must not run on untrusted PR code.

  • Release jobs must not run on untrusted PR code.

    Publishing workflows should run only from trusted main/tag/release events, never `pull_request_target` with checkout of PR code.

  • Exact binary identity may matter.

    If the container must contain the exact same bytes as the release archive, we should wire image build from prebuilt artifacts instead of rebuilding from source.

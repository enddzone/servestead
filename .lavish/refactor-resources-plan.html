Move generated scripts, Compose, and config files into embedded resources without changing behavior.

The repository currently has no standalone shell scripts, Dockerfiles, or Compose files at the project root. The deployable artifacts are generated from Go functions in files like proxy.go, observability.go, network.go, and hardening.go. The refactor should extract the file-like content and larger scripts into a readable resources/ tree, embed them with go:embed, and keep existing call sites stable.

Success criteria
  • Existing CLI commands and remote paths remain unchanged.
  • Rendered Compose, YAML, JSON, env, and shell scripts stay behaviorally equivalent.
  • Resource names and locations are easy to scan and retrieve.
  • Tests cover rendering and existing deployment task composition.

Main Design Choices

Balanced extraction

Recommended

Move file-like artifacts and larger procedural scripts. Keep short command helpers and validation logic in Go.

  • Best readability gain without turning every task into template plumbing.
  • Lower quoting risk than maximal extraction.
  • Matches current tests with small call-site changes.

Maximal extraction

Move nearly every command fragment into shell templates, including small apt and user-management snippets.

  • Most visible shell code in files.
  • More template variables and more surface for quoting mistakes.
  • Likely overkill for one-line commands.

Compose/YAML first

Move only Compose and configuration files now, then revisit scripts later.

  • Lowest first diff risk.
  • Leaves the hardest-to-read shell scripts inline.
  • Good fallback if you want a very incremental PR.

Implementation Plan

1. Characterize current output

Use existing tests as the safety net, then add focused render tests only where coverage is thin.

Verify: go test ./... before resource movement, plus existing Docker Compose validation tests when local Docker is available.

2. Add embedded resource access

Create resources/resources.go with embed.FS and path constants. Add a root-package renderer so existing quote helpers can be reused without import cycles.

Verify: compile-only tests prove every referenced resource exists and every template parses.

3. Move file-like artifacts

Start with static and low-risk files: Docker daemon JSON, sshd hardening config, auto-upgrades config, Beszel config, env file, then Compose/YAML templates.

Verify: current tests that assert service names, labels, image pins, config values, and Compose validity keep passing.

4. Move larger scripts

Extract Pangolin bootstrap, UFW masquerade, swap setup, resource reconciliation, and Dockhand scripts after the renderer is proven.

Verify: tests continue checking key commands, API payloads, CSRF headers, cleanup traps, and credential error messages.

5. Update docs and run full verification

Document the resource organization only if useful for maintainers. Avoid touching user-facing README behavior unless generated artifact details are currently described there.

Risks and Boundaries

Mitigations

Shell and YAML quoting can regress silently. Reuse the existing shellQuote, yamlSingleQuote, yamlDoubleQuote, and jsonString behavior from templates.
Template parse errors should fail during tests or package initialization, not during a half-completed remote deployment.
Existing dirty worktree changes should be left alone. The implementation should touch only resource-related files and tests.

Non-goals

  • No image version changes.
  • No remote path changes under /opt/aegisnode or /etc/aegisnode.
  • No changes to user-owned configuration repository layout.
  • No Dockerfile work unless a Dockerfile is introduced or discovered.
  • No rewrite to YAML encoders or a new deployment abstraction as part of this refactor.

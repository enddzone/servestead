Chosen direction

Two-mode web UI for setup and day-2 operations

Use Guided Workbench for setup because it is safer, calmer, and harder to get lost. Use Operations Console for day-2 work because saved profiles, stacks, GitOps, cloud state, and run history become resources that need filtering, drawers, and fast recovery actions.

Go local web server templ components htmx panels/forms SSE logs/status WebSocket optional
Mode B

Setup Workbench

A guided flow for provisioning, profile intake, repository setup, review, execution, and recovery.

  • Stepper and validation first.
  • Review before billable or destructive actions.
  • Live run dock stays attached to the same page.
Mode C

Operations Console

A resource browser for profiles, stacks, GitOps, runs, secrets access, and cloud actions.

  • Tables, filters, detail drawers.
  • GitOps diff/stage/commit/push/sync in context.
  • Run history and logs are searchable.

System shape

Architecture: Server-owned state, browser-owned interaction

The browser should not own privileged state. It renders fragments and controls, while the Go process keeps authority over profiles, secrets, Git, SSH, DigitalOcean, and setup runs.

Build plan

Phased Implementation

Each phase has a concrete slice, acceptance criteria, and avoids a big-bang rewrite. The existing CLI/TUI can remain while the web UI matures.

0

Web foundation

No product workflow yet. Build the local web runtime and component system.

  • `servestead ui` starts loopback server.
  • One-time browser token and shutdown route.
  • templ layout, CSS tokens, htmx bootstrapping.
  • SSE broker skeleton and error toasts.
1

Guided setup MVP

Replace the most painful setup input/review flows first.

  • Start/profile intent screen.
  • Required values form with server validation.
  • GitOps source + GitHub PAT panels.
  • Review screen before execution.
2

Live run + recovery

Make stage execution feel anchored and observable.

  • SSE logs and status dock.
  • Cancel/retry run controls.
  • Recovery panels for dirty repo, bad domain, missing credentials.
  • Pangolin access reveal after completion.
3

Operations console MVP

Introduce day-2 resource management without the command-center shell.

  • Profiles table + detail drawer.
  • Stacks table + stack drawer.
  • GitOps console with diff/stage/commit/push/sync.
  • Run selected setup/stacks from drawers.
4

Access, cloud, history

Round out the operations surface.

  • Secrets/access page with reveal guards.
  • Cloud restart/destroy/provision flows.
  • Run history, log search, and saved failure summaries.
5

Polish and accessibility

Make it feel fast and robust.

  • Keyboard shortcuts scoped to active region.
  • Focus management after htmx swaps.
  • Responsive layouts for tablet/mobile review.
  • Copy buttons, deep links, empty/error states.
6

Optional enhancements

Only after core workflows are stable.

  • Command palette.
  • TinyGo WASM island for very large log/diff virtualization.
  • PWA install shell if local web becomes the primary UI.
✕

Explicitly not v1

Avoid scope creep while replacing the TUI.

  • No full command-center dashboard yet.
  • No remote multi-user web service.
  • No full Go/WASM app rewrite.

Implementation map

Routes, Fragments, and Streams

HTMX keeps most interactions simple: links and forms request fragments. SSE streams logs and run status into fixed targets.

Shell routes

  • `GET /ui` initial shell
  • `GET /setup/start` workbench start
  • `GET /ops/profiles` operations console
  • `GET /assets/*` embedded static assets

Setup fragments

  • `POST /setup/profile-values` validate values
  • `POST /setup/repository` validate repo source
  • `GET /setup/review?stage=platform` review panel
  • `POST /setup/run/:stage` start run

Operations fragments

  • `GET /ops/profiles/:id/drawer` profile drawer
  • `GET /ops/stacks?profile=:id` stacks table
  • `GET /ops/gitops/:id/diff` diff panel
  • `POST /ops/gitops/:id/commit` commit action

SSE streams

  • `GET /events/runs/:runID` logs + progress
  • `GET /events/profile/:id` profile status changes
  • Events swap `#run-status`, append `#run-log`.

Security defaults

  • Bind `127.0.0.1` by default.
  • One-time token in URL then cookie/session.
  • CSRF token for POSTs.
  • Mask secrets by default; reveal with timeout.

Testing

  • Handler tests for fragments and validation.
  • Golden-ish HTML tests for key screens.
  • Playwright smoke tests for setup/run/GitOps flows.
  • SSE fake run harness.

AegisNode design plan

Git-Backed, Consumer-Owned Compose Configuration

Make Git the source of truth for declarative stack configuration, beginning with the observability Compose file.

Summary

  • Git-backed storage is viable. AegisNode creates initial files, validates them, deploys exact commits, and detects drift.
  • Consumers own tracked configuration. AegisNode must never silently overwrite their edits.
  • Implement a concrete Git workflow first instead of introducing a speculative multi-backend abstraction.
  • Start with stacks/observability/compose.yaml. Migrating the proxy stack remains future work.

Repository and User Interface

  • Use one repository per saved server profile.
  • Add config_repository_path to the profile model while preserving compatibility with existing profiles.
  • Default repositories to $XDG_CONFIG_HOME/aegisnode/repositories/<profile-id>, falling back to $HOME/.config/aegisnode/repositories/<profile-id>.
  • Deleting an AegisNode profile must not delete its repository.
  • Add --config-repo <path> to use or create a local repository.
  • Add --github-repo <https-url> to clone a GitHub repository, optionally into the configured local path.
  • Read private GitHub credentials from AEGISNODE_GITHUB_TOKEN.
  • Interactive setup offers three choices: create a local repository, use an existing checkout, or clone a GitHub HTTPS repository.
  • Initialize new repositories on main and commit the scaffold as AegisNode <aegisnode@localhost>.
  • If a supplied repository lacks the Compose file, write the scaffold and stop before deployment so the consumer can review, commit, and push it.
  • Deploy committed HEAD only. Dirty observability paths block deployment; unrelated working-tree changes do not.
  • Official v1 remote support is GitHub HTTPS only.

Deployment and Ownership Model

  • Store the consumer-owned Compose file at stacks/observability/compose.yaml.
  • Allow edits to images, resource limits, environment, and other settings while enforcing the required AegisNode contract.
  • The required contract includes the beszel, beszel-agent, and dozzle services, the external aegis-public network, stable Pangolin resource labels and targets, and no public Beszel or Dozzle ports.
  • When a newer AegisNode version requires an incompatible contract, fail with migration guidance instead of regenerating the file.
  • For local-only repositories, upload the committed file over SSH and record its commit and file hash on the VPS.
  • For repositories with a GitHub origin, fetch on the VPS and check out the exact local commit. Require that commit to have been pushed first.
  • Use /opt/aegisnode/repository for the remote checkout or snapshot.
  • Keep runtime data under /opt/aegisnode/stacks/observability.
  • Refuse dirty remote checkouts or hash mismatches. Do not discard or import remote edits automatically.
  • Reconcile only during an explicit AegisNode run; do not install a timer, webhook, or daemon in v1.
  • Validate the rendered Compose model before stopping the current stack.
  • Preserve Beszel data and keys during migration. Remove the old secret-bearing Compose file only after the new stack starts successfully, retaining it for recovery if migration fails.

Secrets and Security

  • Replace literal credentials in the tracked Compose file with environment substitutions.
  • Generate /etc/aegisnode/observability.env as a root-owned file with mode 0600, and pass it explicitly to every Compose command.
  • Keep Beszel data, generated configuration, and key material outside Git.
  • Never persist the GitHub token on the VPS or include it in repository URLs, logs, process arguments, commits, or profile files.
  • Send the token transiently over SSH standard input to an askpass flow, then delete all temporary credential material.
  • Recommend a fine-grained token restricted to the selected repository.
  • Extend setup preflight and doctor to require the local Git CLI.
  • Continue using the Git package already installed remotely during bootstrap.

Reference: GitHub personal access token guidance .

Verification

  • Unit-test XDG path resolution, backward-compatible profiles, scaffolding policy, exact-commit selection, dirty-path rejection, GitHub URL validation, token redaction, and secret-free generated Compose.
  • Integration-test local initialization, supplied-repository review flow, GitHub-origin command construction, local snapshot deployment, remote drift rejection, and transition from an uploaded snapshot to a GitHub checkout.
  • Validate generated and consumer-edited files with Docker Compose and verify the required service and Pangolin contract.
  • Confirm migration preserves existing Beszel data.
  • Confirm committed consumer changes are deployed without regeneration.
  • Normal automated tests must not contact GitHub or a live VPS.

Deferred Alternatives

  • Read-only deploy keys for future unattended pulls.
  • SOPS-encrypted Git secrets if secrets later require declarative versioning.
  • GitHub Actions or a systemd timer for push-driven reconciliation.
  • Automatic GitHub repository creation and pushes.
  • Generic Git interfaces after a second remote provider is required.
Planning artifact only. Implementation is intentionally deferred.
